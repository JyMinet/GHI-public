Transparence et intégrité des données

Le Global HashCost Index (GHI) est conçu comme un standard ouvert. Cette page résume les engagements de transparence, la structure des dépôts publics et les limites actuelles du projet.

1. Principes de transparence

  • Code du moteur accessible dans un dépôt public.
  • Documentation méthodologique versionnée et archivée.
  • API publique documentée, avec schémas de réponse et exemples.
  • Versionnage explicite : GHI, méthodologie, moteur, API.

2. Code et dépôts publics

  • ghi-engine : moteur Sandbox, modèles, tests.
  • GHI-public : site web, documents PDF, pages HTML.

Les contributions (issues, suggestions) sont encouragées via les dépôts GitHub, dans le respect des licences et du cadre du projet.

3. API publique et tests

  • Suite de tests automatisés couvrant les endpoints principaux de l’API.
  • Contrôle systématique de la cohérence des champs (min/avg/max, versions, métadonnées).
  • Sandbox Engine v0.4.0 : 17/17 tests unitaires et d’intégration validés.

4. Données Sandbox et avertissements

  • Les données actuelles sont des données factices à but de démonstration.
  • Elles ne doivent pas être utilisées comme estimation officielle du coût réel du Bitcoin.
  • Les futures versions intégreront des données réelles, clairement identifiées comme telles.

5. Gouvernance et évolutions

  • Découplage strict entre :
    • la méthodologie (GHI v1.0.0),
    • le moteur d’exécution (Sandbox Engine v0.4.0),
    • et la couche de présentation (site, dossiers institutionnels).
  • Chaque évolution majeure fait l’objet d’une nouvelle version documentée (changelog, pages dédiées).

6. Contributions et retours

Les institutions, chercheurs et opérateurs industriels intéressés peuvent proposer des retours méthodologiques, des jeux de données ou des pistes d’amélioration via GitHub ou les canaux de contact mentionnés dans les documents officiels.

Transparency and data integrity

The Global HashCost Index (GHI) is designed as an open standard. This page summarizes our transparency commitments, public repositories and current project limitations.

1. Transparency principles

  • Engine code available in a public repository.
  • Methodology documentation versioned and archived.
  • Public API documented with response schemas and examples.
  • Explicit versioning for GHI, methodology, engine and API.

2. Code and public repositories

  • ghi-engine: sandbox engine, models, tests.
  • GHI-public: website, PDF documents, HTML pages.

Contributions (issues, suggestions) are welcome through GitHub, within the scope of the project’s licence and governance.

3. Public API and tests

  • Automated test suite covering the main API endpoints.
  • Systematic checks on field consistency (min/avg/max, versions, metadata).
  • Sandbox Engine v0.4.0: 17/17 unit and integration tests passing.

4. Sandbox data and disclaimers

  • Current outputs are synthetic sandbox data for demonstration only.
  • They must not be used as an official estimate of the real Bitcoin production cost.
  • Future releases will integrate real data, clearly flagged as such.

5. Governance and evolution

  • Strict separation between:
    • the methodology (GHI v1.0.0),
    • the execution engine (Sandbox Engine v0.4.0),
    • and the presentation layer (website, institutional material).
  • Each major change is documented via a new version (changelog, dedicated pages).

6. Contributions and feedback

Institutions, researchers and industrial operators are invited to provide methodological feedback, datasets or improvement ideas through GitHub or the contact channels mentioned in the official documents.
